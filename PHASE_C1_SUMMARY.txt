HOW IT WORKS (Executive Summary)
================================================================================

HARD GATES (Blocks risky tickets → 0% EV):
  • 3+ alt-stats in same parlay
    Example: "Pass attempts" + "Rush attempts" + "Targets" → BLOCKED
  • 2+ early-sequence stats in same parlay
    Example: "Points (First 3 Min)" + "Assists (First 3 Min)" → BLOCKED
  • 2+ event-binary stats in same parlay
    Example: "Dunk" + "Steal" → BLOCKED
  • 2+ same-player stats in same parlay
    Example: "Lamar Pass Yards" + "Lamar Pass Attempts" → BLOCKED

SOFT PENALTIES (Discourages risky combinations → EV multiplier):
  • 2 alt-stats: 95% EV (5% cut)
  • Early + Event mixed: 90% EV (10% cut)
  • 2 same-team: 85% EV (15% cut)
  • 3 same-team: 70% EV (30% cut) + tier downgrade

CORE PICKS (Stat class="core"):
  • Never blocked or penalized
  • Can be freely combined
  • Not affected by Phase C-1

SYSTEM ARCHITECTURE (6-LAYER DEFENSE)
================================================================================

INPUT: picks.json (user enters lines)
  ↓
  Layer 1: STAT CLASSIFICATION (prob.py)
  Assigns stat_class: "core" | "volume_micro" | "sequence_early" | "event_binary"
  ↓
  Layer 2: REGIME GATING (prob.py)
  Checks if stat_class allowed in current game state
  ↓
  Layer 3: CONFIDENCE CAPS (prob.py)
  Applies ceiling: 75% (core), 68% (volume), 65% (sequence), 55% (event)
  ↓
  Layer 4: SAMPLE SIZE + VOLATILITY PENALTIES (calibration.py)
  Shrinks confidence for small samples, high-variance stats
  ↓
  Layer 5: CORRELATION PENALTIES (daily_pipeline.py)
  Reduces EV when picks move together (correlation tagger)
  ↓
  Layer 6: TICKET-LEVEL EXPOSURE ← C-1 (ticket_governor.py)
  Hard gates block dangerous combinations, soft penalties for risky ones
  ↓
  OUTPUT: Cheatsheet with verdicts + penalties displayed

(Planned Layer 7: SLATE-LEVEL EXPOSURE ← C-2, mid-January)

Code Flow:

QUESTION & ANSWER
================================================================================

Q: Why these specific hard gates (3+ alt-stats, etc.)?
A: They represent overconcentration points where correlation becomes problematic.
   3+ alt-stats = overconfidence in derived models. 2+ early-sequence = same
   time window = high correlation. Based on expert judgment; will refine with
   Phase C-2 live data.

Q: Can I override a blocked ticket?
A: Not in C-1 (safety-first). Phase C-2 may add override logic or whitelist
   exceptions. For now, restructure the ticket to pass gates.

Q: What if alt-stats aren't being ingested yet?
A: C-1 won't block anything (all picks are "core" now). Once alt-stats added
   to picks.json, C-1 will activate automatically.

Q: How do I know if C-1 is working?
A: Check cheatsheet for "⚠️ APPROVED (with penalty)" or "❌ BLOCKED" messages.
   If all say "✅ APPROVED", either no alt-stats ingested or all tickets clean.

Q: What's the performance impact?
A: <1ms per ticket (negligible). Daily pipeline overhead: +0.1ms per 156 picks.

Q: Can I disable C-1?
A: Yes, comment out ticket_governor.evaluate_ticket() call in daily_pipeline.py.
   Or set penalty_mult=1.0 in VerdictType.APPROVED fallback.

Q: When is Phase C-2?
A: Mid-January (after 2–3 weeks of C-1 live data). Will add slate-level caps
   and cross-ticket volume limits.
